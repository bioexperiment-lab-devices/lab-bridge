# Обзор системы

## Назначение

Система предоставляет лабораторным исследователям удалённый доступ к лабораторным приборам (насосам, клапанам, денситометрам), которые физически подключены к Windows PC внутри лабораторной сети. Раньше этими приборами можно было управлять только из Python-скрипта, запущенного на том же лабораторном ПК через serial-кабель — эксперименты приходилось писать и запускать на месте. Система снимает это ограничение: теперь исследователь может писать и запускать notebooks с экспериментами из любого места, где есть браузер, работая с реальным оборудованием.

Архитектура состоит из трёх компонентов:

- **[`lab-bridge`](https://github.com/bioexperiment-lab-devices/lab-bridge)** — публичный веб-сервер, на котором развёрнута общая среда для notebooks.
- **[`serialhop`](https://github.com/bioexperiment-lab-devices/serialhop)** — небольшая программа, устанавливаемая на лабораторный ПК и предоставляющая доступ к приборам извне.
- **[`bioexperiment_suite`](https://github.com/khamitovdr/bioexperiment_suite)** — Python-библиотека, которая используется в notebooks для управления экспериментами. Сейчас она мигрирует с локального управления приборами через serial-кабель на удалённое управление через сеть. План миграции описан в [HTTP client design spec](https://github.com/khamitovdr/bio_tools/blob/main/docs/superpowers/specs/2026-04-27-lab-devices-http-client-design.md).

Система уже развёрнута и работает:

- **JupyterLab** (рабочее пространство исследователя): [https://111.88.145.138/lab](https://111.88.145.138/lab)
- **Grafana** (логи lab-агентов и дашборд состояния): [https://111.88.145.138/grafana/](https://111.88.145.138/grafana/)

## Как компоненты связаны между собой

Исследователь открывает среду notebooks в браузере, пишет Python-код с использованием `bioexperiment_suite`, и этот код прозрачно общается с приборами в лаборатории. Лабораторный ПК сам инициирует исходящее соединение к публичному серверу, поэтому лабораторной сети не нужно открывать входящие порты или настраивать port-forwarding — типичное препятствие в институциональных сетях.

## 1. [`lab-bridge`](https://github.com/bioexperiment-lab-devices/lab-bridge) — публичный сервер

`lab-bridge` — это общее рабочее пространство команды, развёрнутое на арендованном VPS (виртуальном сервере) с публичным URL ([JupyterLab](https://111.88.145.138/lab) · [Grafana](https://111.88.145.138/grafana/)). Развёртывается как единый Docker Compose стек, содержащий:

- **JupyterLab** — среда notebooks, в которую логинятся исследователи; защищена общим командным паролем.
- **Caddy** — веб-сервер перед JupyterLab, автоматически обрабатывающий HTTPS.
- **chisel** — сервис туннелирования, позволяющий каждому лабораторному ПК подключаться и публиковать свои приборы внутри среды notebooks.
- **Loki + Grafana** — стек observability, который собирает логи с каждого lab-агента и отображает их в готовом дашборде (live tail, количество ошибок, версия ПО на каждой лабораторной машине).

Развёртывание нового VPS или регистрация новой лабораторной машины автоматизированы небольшим набором команд оператора (`task secrets:add-client`, `task provision`, `task deploy`).

## 2. [`serialhop`](https://github.com/bioexperiment-lab-devices/serialhop) — lab-агент

`serialhop` — небольшая программа, устанавливаемая на Windows PC, к которому физически подключены приборы. Распространяется в виде одного `.exe` файла. После установки работает как Windows service — стартует автоматически при загрузке системы и переживает перезагрузки и выходы из системы. Оператор взаимодействует с ней через небольшую нативную панель управления (Install / Uninstall / Restart / Open config / Open log).

Во время работы выполняет три задачи:

1. **Обнаруживает приборы.** Сканирует serial-порты ПК, определяет, какие из них — насосы, клапаны или денситометры, и кэширует список.
2. **Предоставляет доступ к ним через небольшой REST API.** Три endpoint'а — получить список устройств, выполнить повторное обнаружение, отправить команду конкретному устройству. Сам агент не интерпретирует команды; он просто перегоняет байты между сетью и serial-портом. Сам словарь команд живёт на стороне клиента.
3. **Открывает защищённый туннель к публичному серверу.** Используя chisel, агент дозванивается до `lab-bridge` и просит опубликовать свой REST API обратно в среду notebooks. С точки зрения notebooks API лабораторного ПК становится доступным так, будто это локальный сервис. Этот же туннель также передаёт логи самого агента в центральный observability-стек.

## 3. [`bioexperiment_suite`](https://github.com/khamitovdr/bioexperiment_suite) — Python-библиотека для экспериментов

Это пакет, который исследователи реально импортируют в свои notebooks. Он предоставляет высокоуровневые объекты — `Pump`, `Densitometer`, `Valve` — и более высокоуровневый модуль `experiment/` для сборки целых протоколов.

До недавнего времени библиотека напрямую общалась с serial-портом, что вынуждало исследователей запускать notebooks на самом лабораторном ПК. Текущая миграция заменяет этот транспорт на HTTP-клиент (`LabDevicesClient`), который общается с `serialhop` через туннель. Высокоуровневый API `experiment/` намеренно остаётся нетронутым — меняется только слой под ним — поэтому существующий код экспериментов продолжает работать с минимальными правками.

Код в notebook после миграции выглядит так:

```python
client = LabDevicesClient(port=9001)
devices = client.discover()
for pump in devices.pumps:
    pump.pour_in_volume(5.0)
density = devices.densitometers[0].measure_optical_density()
```

## End-to-end flow

1. Лабораторный ПК загружается. `serialhop` стартует как Windows service, сканирует приборы и открывает туннель к `lab-bridge`.
2. Исследователь — где угодно, лишь бы был интернет — открывает публичный URL, логинится в JupyterLab общим командным паролем и запускает notebook.
3. Notebook создаёт `LabDevicesClient`. Вызов `.discover()` отправляет HTTP-запрос через туннель в `serialhop`, который возвращает список подключённых приборов.
4. Последующие команды — задать скорость потока, налить объём, считать оптическую плотность — идут тем же путём: вызов Python-метода превращается в HTTP-запрос, который превращается в serial-команду на лабораторном ПК, а ответ возвращается обратно тем же маршрутом.
5. Всё это время `serialhop` стримит свои логи на центральный сервер, где оператор может смотреть их вживую в Grafana.

## Сводка по технологиям

| Слой | Технология |
|---|---|
| Публичный сервер | Docker Compose со связкой Caddy, JupyterLab, Loki и Grafana, развёрнутый на VPS |
| Туннелирование | chisel (лабораторный ПК дозванивается наружу — входящие порты не нужны) |
| Lab-агент | Go, распространяется как один Windows `.exe`, работающий как service |
| Библиотека для экспериментов | Python (`httpx` для HTTP, `loguru` для логирования) |
| Протокол приборов | Кастомные serial-команды на уровне байт, определены отдельно для каждого типа устройства |
